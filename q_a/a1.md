## Q1 — Diagrams and visual communication

### How important are diagrams vs prose on dev.to and Zenn?

Important, and more so than in long-form English tech blogging generally —
but as **structure carriers, not argument carriers**. Both platforms have
skim-heavy reading cultures: readers scroll first, then decide whether to read.
Prose carries the argument; a diagram carries the *shape* of the argument, and
the shape is what a skimming reader evaluates. Concretely:

- **Zenn** renders Mermaid natively in markdown (```` ```mermaid ```` blocks) and
  the JA technical-writing culture there leans on figures and tables heavily;
  an architecture article without a single figure reads as unfinished.
- **dev.to** does *not* render Mermaid; it needs hosted images (or ASCII in code
  blocks). Cover images drive clicks; in-article diagrams drive completion.
- For the frameworks you have: F1 (project intro) and F2 (engineering lessons)
  benefit most from one **overview diagram**; F3 (evaluation methodology) and
  F4 (survey) benefit most from **comparison tables** — which are markdown,
  cheap, and portable, so favor tables over diagrams whenever the content is
  comparative rather than topological.

One good visual near the top outperforms three mediocre ones scattered through.
The failure mode to avoid is decoration: a diagram that restates a heading.

### Should the assistant actively propose new visuals?

**Yes — propose, never insert.** This is the same philosophy the plugin already
commits to everywhere else: harvest routes unsourceable claims to NEEDS-OWNER
instead of asserting them; review emits findings instead of rewrites; you
arbitrate. Visuals should follow the identical contract: the assistant
*proposes* a visual with a stated rationale ("this section describes a 4-step
flow; a sequence diagram would let a skimmer grasp it"), and the author
accepts, modifies, or declines.

Critically, **diagrams are claims**. An architecture diagram asserts that
component A calls component B. Every element in a proposed visual must
therefore be source-pointed exactly like a fact-sheet entry, or the proposal
carries a `[VERIFY]` marker. A diagram invented from vibes is worse than an
invented sentence, because readers grant diagrams more authority.

### Framework-slotted or opportunistic?

**Both, with different mechanics:**

- **Framework slots** (deterministic): each framework declares its expected
  visuals — F1: one overview diagram slot; F2: optional before/after or
  timeline slot; F3: one comparison table slot (required — an evaluation
  article without a results table is broken); F4: one landscape table or
  concept-map slot. A slot the author declines is filled with nothing, not a
  placeholder.
- **Opportunistic proposals** (bounded): during framework fill, the assistant
  may flag up to N additional sections (suggest N=2) where a visual would
  materially help, with rationale. Bounded, because unbounded suggestion lists
  are review-fatigue — the same reason the gap interview is capped at 5
  questions.

### The fallback ladder when the repo has no suitable diagram

In order of preference:

1. **Reuse a repo diagram** (what F1 already did with the README ASCII diagram)
   — authoritative, source-pointed, zero cost. Always first.
2. **Generate Mermaid** — text-native, diffable, renders on Zenn and GitHub,
   and reviewable in the draft itself. This is the workhorse.
3. **Generate a figure spec** (what to show: elements, relations, emphasis,
   caption) — when the visual is beyond Mermaid's vocabulary (rich
   illustrations, screenshots-to-be-taken, plots of real data).
4. **Generate an image-generation prompt** from the figure spec — see below.
5. **ASCII** — only when the target context can't render anything else and the
   structure is simple. ASCII degrades badly with complexity.
6. **`[Figure: …]` placeholder** — never emitted alone; a placeholder is just a
   figure spec with the spec deleted. If the ladder reaches this rung, emit the
   figure spec (rung 3) instead.

PlantUML is excluded: it requires a rendering server or Java toolchain, and
Mermaid covers the needed diagram types.

**Platform variants (pipeline stage 5) handle rendering divergence:** the Zenn
variant embeds Mermaid source directly; the dev.to variant carries the Mermaid
source in an HTML comment plus an explicit publish-blocker line ("render this
to an image before publishing") — because the plugin's no-JS constraint means
it must not bundle `mermaid-cli`, and rendered-image generation is the owner's
tooling per the pipeline spec's existing non-goal. This also fits the
completion-summary finding already in the dogfood log: "render figure X" is a
*publish blocker*, cleanly categorized.

### Copy-paste-ready image-generation prompts

**Yes, generate them automatically — this is exactly the kind of mechanical
work the assistant should absorb.** When the ladder lands on rung 4, the
assistant emits a fenced, self-contained prompt block: subject, style
constraints, aspect ratio, text-in-image warnings (image models butcher
embedded text — the prompt should say "no text labels" and let the author add
labels in post, or accept Mermaid instead), and the caption to use. Writing
these by hand is unnecessary work; reviewing one is seconds.

### Who decides whether a diagram is worth adding?

**The author, always — but the assistant must make the decision cheap.** You
said you don't know the dev.to/Zenn reading culture well enough to judge; the
assistant does, so it should *brief* you: propose with a one-line rationale and
a preview (the Mermaid source or figure spec), and you approve/decline exactly
like a gap-interview question. Note the dogfood findings you've already logged
(approval choices must state their concrete effect; show context before asking)
apply verbatim to visual proposals — build that in from day one rather than
re-learning it.

→ **SPEC-article-visuals** in §Specs. Note it must *amend*
`SPEC-article-draft-pipeline`'s non-goal ("No image/figure generation") — per
your own constraint, behavioral changes to adopted specs go through spec
updates first. The amendment is narrow: *proposing* visual source/specs/prompts
is in scope; *rendering* images remains out.

## Q2 — Article length

**Ignore length as a target. Optimize for completeness-without-padding, and
report length as information, never as a constraint.**

- **Technical-writing view:** good technical articles are *dense*, not short or
  long. The correct length is an emergent property of the framework: every slot
  filled, no slot padded. The frameworks are already your length control — they
  bound *structure*, which bounds length far better than a word count does. A
  word-count target corrupts writing in both directions: padding to reach it,
  or amputating necessary caveats to fit it. Your instinct ("good writing
  deliberately avoids thinking about word count except platform limits") is
  correct; adopt it as policy.
- **DX view:** the one thing worth surfacing is *reading time* (words ÷ ~200wpm
  EN, chars ÷ ~500cpm JA) in the completion summary as an informational note —
  it helps you calibrate expectations across articles, and costs nothing. It
  belongs in the "informational" bucket of the completion-summary
  restructuring you already logged as a finding. This is a one-line addition to
  the existing zero-token lint script, not a capability.
- **Product-evolution view:** resist adding length knobs even later. If a
  platform ever imposes a hard limit, that's a *validation* (a publish blocker
  in review), not an optimization target. The review structural pass already
  proposes cuts where prose is bloated — that's the correct mechanism, because
  it cuts for *redundancy*, not for count.

**No implementation spec — deliberately.** The recommendation is to make
"length is not a target" an explicit line in the frameworks spec's conventions
(one-sentence spec edit), and add the reading-time metric when the
completion-summary finding gets fixed. Neither warrants a spec document.

## Q3 — Review philosophy

**Keep V1 exactly as scoped: findings, not rewrites, author's intent
untouched. Your proposed alternative is real but it is not review — don't
grow review to cover it.**

The five things you list (contradictions, readability, reasoning gaps,
structure, reader's perspective) are all *intent-preserving*: they improve how
well the article achieves the story the author already chose. The case you
describe — "after reading the draft I decide a section should be reordered,
merged, removed, or rewritten because it no longer matches my intended story" —
is *intent-changing*: the story itself moved. Those are different operations
with different contracts:

- Review answers: "does the article achieve its intent?" → findings.
- Restructure answers: "the intent changed; re-derive the article." → a new
  outline, then a mechanical re-fill.

Growing review into restructure would break review's best property — bounded,
capped, arbitratable findings — because "the story is wrong" is not a finding,
it's a re-brief. The correct home for restructure is the *draft pipeline*
(it's a re-entry into framework fill with a changed outline), not the review
workflow.

**Should you build it now? No.** Your own instinct is the right one, and it's
also this project's own stated philosophy (the dogfooding gate): keep V1
simple, manually edit after review, and add richer workflows only when real
usage demonstrates need. Make the gate *concrete* so it isn't a vibe:

> **Trigger:** when `docs/dogfood-findings.md` records **3 or more** article
> runs where post-review manual editing involved moving/merging/removing whole
> sections (not sentence-level edits), build SPEC-article-restructure.

Until then, log each occurrence in the findings log — you already have the
habit and the file. The one thing worth doing *now* is writing the spec while
the design intuition is fresh, marking it deferred. It's included in §Specs
with the trigger stated in its frontmatter, so the decision is
pre-made and the future work is a mechanical "the trigger fired."

Design sketch the spec encodes (so review of it is cheap): the author states
the new intended story in 1–3 sentences → the assistant produces a **re-outline
proposal**: a table mapping every existing section to keep / move / merge /
drop / rewrite, with rationale — findings-style, author arbitrates → on
approval, mechanical re-fill that **preserves all source-pointed facts and
their pointers** (facts survive restructuring; only arrangement changes), and
any newly needed content routes through the normal gap-interview mechanism,
not open-ended generation. One restructure per draft version, same as review
passes — a second restructure means the story is still unsettled and the fix is
thinking, not tooling.

## Q4 — Long-term article management

**Articles staying in their target repos is correct today and stays correct at
year five. The future capabilities you list don't need the articles — they
need *metadata about* the articles. Add an index; never move content.**

Why per-repo storage remains right from all three perspectives:

- **Technical writing:** an article's authority comes from proximity to its
  sources. The draft was harvested from that repo at a commit; keeping it
  there keeps every source pointer resolvable forever. Centralizing articles
  would orphan them from their evidence.
- **DX:** the plugin's whole architecture (CAP-1: runs *inside* the repo it
  writes about) depends on this. A central article store would force every
  writing session to span two repos.
- **Product evolution:** every capability you list — avoid repetition, detect
  cross-article contradictions, track coverage, suggest follow-ups, build an
  article graph — is a query over *article metadata* (topics, claims,
  publication status, source repo/commit), not over article prose. Queries
  need an index, not a warehouse.

So the architecture is: **content in host repos (unchanged), plus one
machine-global metadata index whose location lives in `user-config.yaml`** —
the same identity/engine split the plugin already has. Each entry is a small
record: slug, title, platform(s) + canonical URL, framework used, topic tags,
key claims (one-liners), source repo + commit, date. Written at the end of a
pipeline run (with author approval, like everything else), read at the start
of the next one ("you covered X in article Y — reference it instead of
re-explaining" / "three articles mention benchmark Z with different numbers —
contradiction?").

Two placement options, and my recommendation:

1. **A dedicated `articles-index` repo (or a directory in your website repo)**
   — recommended. Git-versioned, greppable, survives machine changes, and the
   website repo is already the publishing hub so the index naturally lives
   near the publication records.
2. `~/.config/writing-assistant/` — simpler but unversioned and single-machine;
   wrong for years-scale data.

**Build it when it hurts, not now.** Same discipline as Q3: the index earns
its complexity at roughly article #5–10, when "am I repeating myself?" becomes
a real question. Concrete trigger:

> **Trigger:** when 5 articles have been published via the pipeline, or the
> first time you catch yourself re-explaining a concept a previous article
> covered — whichever comes first — build SPEC-article-index.

The spec is in §Specs, deferred, trigger in frontmatter. One design decision
is pre-made there because it's load-bearing: index entries store *pointers and
one-line claims*, never article bodies — the same pointers-over-copies
principle as the fact sheet, and it keeps the index mergeable and tiny.

```markdown
# Article Visuals

## Why

Skim-heavy platforms (dev.to, Zenn) evaluate an article's shape before its
argument; one well-placed structural visual materially improves completion.
Today the pipeline reuses repo diagrams when they exist (good) but is
forbidden from producing anything else, so articles whose repos lack diagrams
ship visually bare, and the owner hand-writes image-generation prompts —
mechanical work the assistant should absorb. Diagrams are claims: this spec
extends the plugin's existing sourcing discipline (source pointers, [VERIFY],
owner arbitration) to visual content rather than inventing a parallel regime.

## Capabilities

- **CAP-1** (framework visual slots)
  - **intent:** Each framework declares expected visuals as slots: F1 one
    overview diagram; F2 optional before/after or timeline; F3 one comparison
    table (required); F4 one landscape table or concept map. A declined slot
    is omitted entirely, never left as a placeholder.
  - **success:** A produced draft's visuals match the framework's declared
    slots; a draft with a declined slot contains no placeholder residue.
- **CAP-2** (proposal, not insertion)
  - **intent:** During framework fill, the pipeline proposes each slot visual
    — and at most 2 opportunistic extras — as an arbitratable item: rationale,
    preview (Mermaid source, table, or figure spec), and choices stating their
    concrete effect on the article. The author approves, modifies, or declines.
  - **success:** No visual enters a draft without an explicit approval; every
    proposal shows rationale + preview + concrete-effect choices.
- **CAP-3** (sourced visuals)
  - **intent:** Every element of a proposed visual is source-pointed like a
    fact-sheet entry or the proposal carries [VERIFY]; unverifiable structural
    claims route to NEEDS-OWNER, same partition rule as prose.
  - **success:** Auditing any approved diagram element leads to a source
    pointer, an interview answer, or a [VERIFY] marker; no exceptions.
- **CAP-4** (fallback ladder)
  - **intent:** When no repo visual fits, generate in strict order: Mermaid →
    figure spec (elements, relations, emphasis, caption) → copy-paste-ready
    image-generation prompt derived from the figure spec (incl. "no embedded
    text" guidance and aspect ratio) → ASCII (simple structures only). A bare
    `[Figure: …]` placeholder is never emitted.
  - **success:** Every non-reused visual in a draft is one of: Mermaid source,
    figure spec, image-gen prompt block, or ASCII; zero bare placeholders.
- **CAP-5** (platform-variant rendering)
  - **intent:** Stage-5 variants handle divergence: Zenn variant embeds Mermaid
    directly; dev.to variant carries Mermaid/figure-spec in an HTML comment
    plus an explicit publish-blocker line ("render to image before
    publishing") in the completion summary's blocker bucket.
  - **success:** The Zenn variant renders its diagrams with zero manual work;
    the dev.to variant's completion summary lists each unrendered figure as a
    publish blocker.

## Constraints

- Proposing visual *source* (Mermaid, figure specs, prompts) is in scope;
  *rendering* images is not — no bundled mermaid-cli or image tooling (no-JS
  constraint; rendering is owner tooling, per SPEC-article-draft-pipeline).
- Prefer markdown tables over diagrams whenever content is comparative rather
  than topological.
- Mermaid only; no PlantUML (rendering-server/Java dependency).
- Proposals obey the logged dogfood UX findings: show where in the outline the
  visual lands, why it is proposed, and what each choice does to the article.

## Non-goals

- No image rendering or hosting; no calls to image-generation APIs.
- No per-article visual style system; captions and alt text only.
- No retrofitting visuals into already-published articles.

## Success signal

A dogfooded F1 article on a repo with no usable diagrams ships with one
approved overview visual (Mermaid on Zenn, blocker-flagged image for dev.to),
every diagram element source-pointed, and the owner spent under a minute per
visual decision.
```

```markdown
# Article Restructure

## Why

Review is intent-preserving by contract: findings, not rewrites, within the
author's chosen story. Dogfooding may surface a different need — the author
reads the draft and the *story itself* changes (sections should be reordered,
merged, dropped). Stretching review to cover this would destroy its bounded,
arbitratable-findings property; the correct home is a re-entry into the draft
pipeline with a changed outline. This spec exists now, deferred, so the
build decision is pre-made and fires mechanically on evidence.

## Capabilities

- **CAP-1** (re-brief)
  - **intent:** The author states the new intended story in 1–3 sentences;
    the workflow accepts no other input form (no open-ended editing session).
  - **success:** Every restructure run starts from a recorded re-brief; runs
    without one are refused.
- **CAP-2** (re-outline proposal)
  - **intent:** The workflow emits a proposal table mapping every existing
    section to keep / move / merge / drop / rewrite with rationale —
    findings-style, author arbitrates each row before anything changes.
  - **success:** No section is altered without an approved mapping row; the
    proposal covers 100% of existing sections.
- **CAP-3** (fact-preserving re-fill)
  - **intent:** On approval, a mechanical re-fill rearranges content per the
    mapping, preserving all source-pointed facts and their pointers verbatim;
    newly needed content routes through the gap-interview mechanism (≤5
    questions), never open-ended generation.
  - **success:** Diffing fact-sheet pointers before/after shows zero lost or
    altered source pointers for kept content; all new claims are
    interview-sourced or [VERIFY]-marked.

## Constraints

- At most one restructure per draft version (mirrors review's once-per-version
  passes); a second requested restructure halts with "the story is unsettled —
  resolve intent before tooling can help."
- Restructure runs between review cycles, never concurrently with one.
- Consumes framework templates verbatim; a restructure cannot change the
  chosen framework (that is a new article).

## Non-goals

- No prose-quality improvement (review's contract).
- No merge of multiple articles into one, or split into several (new-article
  operations).

## Success signal

On a real draft where the owner's story changed post-review, the restructure
completes in ≤5 minutes of owner attention, the re-filled draft passes
fact-pointer diffing with zero losses, and the owner hands it back to review
instead of manually rebuilding sections.
```

```markdown
# Article Index

## Why

Articles correctly live in the repos they were harvested from — proximity to
sources keeps every pointer resolvable. But years-scale capabilities (avoid
repetition, cross-article contradiction detection, coverage tracking,
follow-up suggestions, an article graph) are queries over article *metadata*,
not prose. A single machine-global index of pointers-and-claims — never
content — serves those queries while leaving the storage architecture intact.

## Capabilities

- **CAP-1** (index record)
  - **intent:** At the end of a pipeline run, with author approval, append one
    record to the index: slug, title, platform(s) + canonical URL, framework,
    topic tags, key claims as one-liners, source repo + commit, date. Records
    contain pointers and one-line claims only — never article bodies.
  - **success:** Every published pipeline article has exactly one record;
    grepping the index for article prose returns nothing.
- **CAP-2** (index location via user config)
  - **intent:** The index location is a key in user-config.yaml (recommended
    target: a directory in the owner's website/publishing repo, git-versioned);
    the engine has no default path and never assumes one.
  - **success:** Changing the config key relocates all index reads/writes with
    zero skill edits; absent the key, index features are silently skipped and
    the pipeline runs exactly as pre-index.
- **CAP-3** (harvest/draft consultation)
  - **intent:** Pipeline start consults the index read-only and surfaces:
    prior-coverage notes ("article Y covered X — link instead of
    re-explaining"), potential claim contradictions, and follow-up candidates
    — as informational proposals the author may ignore.
  - **success:** A run on a topic adjacent to an indexed article surfaces the
    overlap before framework fill; ignoring every suggestion changes nothing
    downstream.

## Constraints

- Optional by construction: no index, no behavior change (CAP-2). The plugin
  must remain fully functional for a user who never configures one.
- Append-only record format, one file per article or one flat file —
  greppable plaintext (YAML/markdown), no database.
- Claim entries are one line each; contradiction detection compares claim
  lines, not article content.

## Non-goals

- No article content storage, mirroring, or syndication from the index.
- No automated topic selection or backlog management (standing non-goal).
- No cross-user/team index in v1.

## Success signal

With ~10 indexed articles, starting a draft on an overlapping topic surfaces
the prior article before the interview stage, and the owner links back instead
of re-explaining — observed in a real run, not a test.
```
