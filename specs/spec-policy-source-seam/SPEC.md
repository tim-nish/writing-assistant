# Policy-Source Interview Seam (A1)

## Why

An opportunity to capture, ratified 2026-07-14: the owner keeps an authoritative policy hub (glossary, lessons, topic positions) in a separate private repository, and the Stage-2 gap interview currently asks generic questions blind to those positions. The hardest interview input is the *right question*; reading the recall surface lets the tool generate questions specific to the owner's recorded positions — surfacing tension (contradictions, reversals, missing rationale), never confirmation. The seam is a local read-only pointer: writing-assistant reads a bounded slice of that policy hub, the owner decides everything, and contribution back is proposal-only. Build order is ratified: this input side ships first; the review-side consistency pass (A2) later reuses the same plumbing.

## Capabilities

- **CAP-1** — `policy_source` config key (amended 2026-07-18, #366 — presence toggle, no filesystem path)
  - **intent:** A host repo may declare an optional `policy_source` block in `writing-sources.yaml`: **`enabled: true`** — a presence toggle only. ~~a local filesystem `path` to the policy-hub checkout~~ *(the `path` key is retired — the consumer never holds the hub's filesystem location; the gateway's own operator config owns it; a leftover `path` key is a named configuration error, migration-noticed)*; ~~an optional `track` … and an optional explicit `topics:` list~~ *(removed — SPEC-policy-topic-at-draft, Story 13.36)*. **Amended 2026-07-21 (#525):** the block may carry an optional `track_topics` **mapping** — articles-repo **track name** → hub **topic name(s)** — that seeds the per-article ≤2-topic proposal (SPEC-policy-topic-at-draft CAP-5). Declared precedence: track names are owned by the **articles repo** (its backlog `track:` frontmatter is the API — no new upstream schema artifact, honoring the deferral at topics/articles.md:20); hub **topic files are authoritative** for topic existence, and on any mismatch the hub wins and the consumer mapping is the defect. No URL scheme, no sync protocol — **the MCP registration is the integration**.
  - **success:** A repo with the block gets policy-seeded interviews served through the gateway; a repo without it behaves exactly as today; `validate-config` reports a malformed block (a leftover legacy `path`/`track`/`topics` key) as a stage-0 configuration error naming the key and fix. For a `track_topics` mapping (#525): a mapped value that resolves to **no hub topic file** is a stage-0 configuration defect (topic-existence lint — the same lint shape used for variant-profile target directories), and a mapping **track** absent from every articles-repo backlog `track:` value is a **warning** (a stale mapping), not an error; an absent mapping is valid and changes nothing.
- **CAP-2** — bounded, pinned, read-only policy reader (amended 2026-07-18, #366 — served transport, same CLI contract)
  - **intent:** The reader keeps its CLI contract (`pin` / `list-topics` / `read [--only|--topics]` / `whitelist`) but is a **tsurezure-gateway MCP client** (consumer `writing-assistant`), never a filesystem reader: the bounded surface — **only** `GLOSSARY.md`, `LESSONS.md`, and ≤2 `topics/*.md` files *(selected per article at draft time — SPEC-policy-topic-at-draft CAP-2; same ≤2 cap)* — is **served** by the gateway, whose realm grants enforce the whitelist server-side; the pin `<policy-source>@<commit>` and every `file:line@commit` cite come from gateway response payloads. The reader opens no file under the hub, holds no hub path, and never writes outside the host run workspace. A gateway **miss** is a served answer (surface it with the question — distill-bug signal); gateway **unavailability** degrades to generic mode with one line; **direct-file fallback is prohibited in both cases**. The gateway's server-side access log is the canonical access record.
  - **success:** No writing-assistant code path opens a recall-surface file directly (a regression test asserts this); every emitted policy quote carries `file:line@commit` from a gateway payload (harvest pointer convention); the run artifact records the pin; nothing under the hub is ever created or modified; a migrated end-to-end run's reads appear in the gateway access log under `consumer=writing-assistant` and match the run's `consulted:` receipts; with the gateway stopped, the same run completes in generic mode with zero hub filesystem reads.
- **CAP-3** — schema-enforced, question-shaped interview items
  - **intent:** Every Stage-2 candidate question is emitted as an interview item `{id, gap_type, seed: {quote, pointer, companion?} | null, question, owner_answer: ""}` (companion `seam-formats.md`). `gap_type` extends the NEEDS-OWNER taxonomy with four tension types — `contradiction`, `ambiguity`, `missing-rationale`, `reversal-candidate` — the only types a policy seed may generate. A validator rejects: a pre-filled `owner_answer`, a tension-typed item without a seed, a seed without a pinned pointer, and a seeded question that merely restates its seed (confirmation is not a gap type).
  - **success:** Each rejection class has a seeded test fixture that the validator fails; a valid item set passes; validation runs before triage so a bad item can never reach the owner.
  - **staleness routing** (added 2026-07-17, #306): a seed that **predates the material it appears to contradict** — decided from inputs the run already holds (the surface's `updated:`/`state:` lines and the run's pin; no new metadata is required of the hub) — is a *stale recorded position*, not a live conflict. Such an item is routed to `gap_type: reversal-candidate` and asks the owner to **confirm or update** the recorded position; it is never presented as a live tension to adjudicate. Its staging candidate (CAP-4) is framed as a **policy-update proposal for the stale line**, never as the resolution of a conflict — otherwise an owner answering a staleness artifact contributes a "resolution" to a dispute that never existed, and the seam's own recall surface degrades.
  - **whole-surface authoring** (added 2026-07-17, #299): a tension item is authored against the **consulted surface as a whole**, never a single quoted line. When another line in the same consulted surface already resolves the apparent conflict, the item is either **not raised** or raised **with the resolving line** carried in `seed.companion` (`{quote, pointer}`, same pinned-pointer grammar) so the owner arbitrates only the residual question. An apparent contradiction the surface itself answers is a *manufactured* tension — it spends a bounded owner-gate slot re-deciding settled ground, and an answer to it contributes a resolution to a conflict that never existed. The distinction such seeds most often miss is stated where items are authored: **harvest assembles evidence; the interview is the judgment gate** — evidence assembly is not unattended generation.
- **CAP-4** — staging-candidate emitter (proposal-only contribute-back)
  - **intent:** When an interview answer contains a durable decision or reversal, the run output appends a staging-candidate block whose schema mirrors the policy hub's staging-area frontmatter (`slug, created, source_repo, perishable, tags` + question/decision in full sentences; companion). The tool stops there — the owner copies accepted blocks into the policy hub by hand.
  - **success:** A run whose answers contain a reversal emits a schema-valid block in its run output; no run ever writes a file under `policy_source.path`.
- **CAP-5** — `consulted:` auditability line
  - **intent:** The interview run artifact ends with an `/ask`-style `consulted:` line naming which glossary entries, lessons, and topic lines seeded which questions (companion format).
  - **success:** For every policy-seeded question asked, the `consulted:` line maps its seed pointer to its question id; a generic-mode run emits `consulted: none (policy_source unavailable|unset)`.
- **CAP-6** — graceful degradation
  - **intent:** An absent `policy_source` key runs the interview in generic mode silently; a present-but-unusable one (path missing, unreadable, or not a git repo) logs one line and degrades to generic NEEDS-OWNER gap questions. The seam adds capability, never fragility.
  - **success:** Deleting or corrupting the policy checkout between runs changes the interview from seeded to generic with one logged line and zero failures; no stage aborts.
- **CAP-7** — policy-result classification (added 2026-07-18, #365)
  - **intent:** Before any owner question is presented, the served policy result for that question's subject is classified into exactly one of four classes, and the presentation follows the class:
    - **determined** — a directly served ratified policy line uniquely answers a **ratified fact** (a recorded decision about topology, naming, architecture — never a judgment): the question is **not asked**; the applied answer is recorded with its citation (`file:line@commit` at the run's pin) in the journal, exactly as a suppressed question records its covering entries.
    - **constrained** — served policy rules out some answers to a ratified-fact question without determining one: the question is still asked, and each **excluded candidate is SHOWN, marked excluded**, carrying the **governing line quoted verbatim with its `file:line@commit` pointer at the run's pin**; the **override remains available** (choosing an excluded candidate is permitted, and — like any reversal of a served ratified line — routes to the staging candidate as a *proposed policy change*, never to "current policy"). The excluded candidates and the excluding lines are journaled **in addition**; journaling is a record, never the presentation. **Silent suppression is a defect of this CAP** (amended 2026-07-22, #566): presenting only compatible candidates makes a constrained question indistinguishable from a free choice, hides the exclusion from audit, and is the same failure as a defaulted multi-outcome gate.
    - **open** — policy does not answer: ask normally (today's behavior, unchanged).
    - **conflict** — served policy, repository state, or authoritative user config **disagree** on the subject: present an **explicit reconciliation question** (conflict-typed item carrying every disagreeing position with its pointer — seam-formats.md), never an ordinary content-preference question whose candidates smuggle the conflict in.
    The **ratified-fact vs owner-judgment line is this CAP's boundary**: owner judgment (opinion, significance, surprise, tradeoff, warning, audience framing) is never pre-decided — for judgment questions the "questions only" rule stands whole and no class other than `open`/`conflict` may apply. An owner answer that **reverses** a served ratified line is recorded as a **proposed policy change** (staging candidate, CAP-4) and is never treated as current policy by the same run's later stages. Nothing under the policy hub is ever written (unchanged).
  - **success:** Replaying the 2026-07-18 EN-topology case (served records-only line vs `syndication.policy` EN-canonical) yields a `conflict` reconciliation question — not a content question offering the records-only position as an ordinary candidate; a question whose subject a served line uniquely determines is absent from the interview with its applied answer + citation journaled; a judgment question is never suppressed or filtered by this CAP; validator fixtures cover each class and the reversal-to-staging route. **A `constrained` fixture presents its excluded candidate visibly** — marked excluded, with the excluding line's quote and pointer — and selecting that candidate is accepted and routed to staging; a fixture that omits an excluded candidate from the presented item **fails** (added 2026-07-22, #566).

## Constraints

- Policy reading may **propose** — tension questions, article-type/topic recommendations, story structures, hub-element candidacy, editorial defaults — under four load-bearing invariants (amended 2026-07-24, #670; stated in full in the amendment note above): **no facts** (a policy line never becomes a SOURCE pointer or grounds a checkable claim — preserves SPEC-article-draft-pipeline CAP-2 and `docs/interview-architecture.md` D1; `verify-provenance` is the backstop), **proposes never decides** (owner-ratified, declines recorded in the presented-payload log), **audited influence** (verbatim quote + `file:line@commit` + a journal `consulted:` line), and **owner judgment never pre-decided** (opinion/significance/surprise/tradeoff/warning/framing stay `open`/`conflict` — consumer NFR15 retained and scoped to judgments). The prior "questions only, with two bounded exceptions" framing is **superseded**: the editorial-judgment default (SPEC-policy-editorial-direction CAP-6), the CAP-7 `determined` ratified fact, and CAP-1's article-type recommendation (Story 13.37) are now **instances** of invariant-2 proposal, not carve-outs. Triage (suppress/recommend/open) and recommendation generation remain a view over harvest output — the policy source never supplies or pre-fills a **factual** answer to an asked question.
- Policy-seeded questions obey the owner-facing proposal contract unchanged: selective prompts, Where/Why/Effect, journal entries; the seed quote + pointer is presented as the question's Why context, and the journal records seed pointers like recommendation groundings.
- The whitelist, the ≤2-topics cap, and read-only-ness are code-enforced (reader takes an allowlist), never prompt-enforced.
- Reader and validator are stdlib-only Python in `scripts/` (repo convention: no PyYAML, no JS/TS); each ships with a `check-*.sh` harness like every other script.
- Seed pointers use the existing `file:line@commit` harvest convention — no new pointer format.
- The policy-hub checkout is never modified, and nothing from its history archive (including its staging area) is ever read.

## Non-goals

- A2 — the review-article policy-consistency pass (ships later as this plumbing's second consumer).
- Automated contribute-back: no writes into the policy hub, no PR automation; the manual copy is the design until it proves to be real friction.
- Sync/export pipelines, URL/remote policy sources, embeddings or semantic search over policy or history, cross-repo `/ask` or any invocation mechanism.
- Redesigning Stage-2 triage, recommendations, or the interview journal beyond the additive fields named here.
- Machine-**final** self-answering of a NEEDS-OWNER question — auto-committing an owner position without ratification — **declined 2026-07-19 (#423)**: the owner ratifies every candidate; a question the served recall surface covers is recorded as an FYI citation under CAP-7's *determined* class, never a silently-adopted answer to a judgment gap. Drafting 1–3 candidate answers from policy is sanctioned (CAP-2 / SPEC-policy-editorial-direction CAP-6); only machine-finality is out of scope.

## Success signal

A draft-article run in a host repo pointing at the policy hub asks at least one tension-typed question quoting a pinned policy line the owner recognizes as their own position — and the same run with `policy_source` removed completes identically minus the seeded questions.
