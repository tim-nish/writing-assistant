# Writing Assistant plugin repository

## Why

An opportunity to capture plus a vision to realize: the article specs already produced in this repo (frameworks, draft pipeline, review) are valuable beyond the website repo, but as `.claude/skills` candidates inside one repo they can serve only that repo. Extracting them into a standalone repository packaged as a Claude Code plugin makes the assistant installable per-repository — so drafting an article about QuantScenarioBench runs *inside* QuantScenarioBench with direct access to its specs, docs, and git history (which is where the draft pipeline's harvest step needs to be). The engine/identity split keeps an open-source release cheap if dogfooding on research-notes proves the tool out.

## Capabilities

- **CAP-1**
  - **intent:** The assistant is a standalone git repository installable as a Claude Code plugin on a per-repository basis; the repo serves as its own marketplace via `.claude-plugin/marketplace.json` (`/plugin marketplace add <owner>/<repo>` + `/plugin install`), with Community Marketplace listing an optional later step for discoverability only.
  - **success:** Installing the plugin in a fresh clone of QuantScenarioBench makes the drafting/review commands available there without copying files into that repo and without any marketplace submission or review.
- **CAP-2** (amended 2026-07-17, #301 — typed sources)
  - **intent:** A harvest step gathers facts from the host repo and any additional sources declared in a per-repo `writing-sources.yaml`. Source entries carry a `type` — `path` (the default; local paths, sibling repos such as research-notes), `github-issues` (the host repo's issues, optionally label-filtered), or `tanuki-den` (read-only access to the target's findings ledger) — with every fact carrying a resolvable source pointer: file/line pinned to a commit, a commit, a URL (the native form for issue facts), or `den:<ledger-id>@<run>` for Den findings. Recurrence counts and dispositions on Den/issue facts are data the pipeline may quote, never judgments it may amplify; the reads are one-way — harvest never writes back to the Den or to issues.
  - **success:** A harvest run against QuantScenarioBench with research-notes declared as a source produces a fact sheet whose every entry resolves to an exact location in one of the two repos; undeclared repos, undeclared source types, and any producer state not named by a declared entry are never read.
- **CAP-3** (amended 2026-07-11 per `docs/harness-architecture.md`)
  - **intent:** The draft pipeline (SPEC-article-draft-pipeline: harvest → ≤5-question interview → framework fill with `[VERIFY]` markers → article-quality gate → platform variants) runs as a plugin skill inside the host repo.
  - **success:** The pipeline's existing success criterion holds when run from a non-website repo: draft in ≤10 minutes of owner attention, zero unmarked invented claims, and the draft passes the article-quality gate (SPEC-article-draft-pipeline CAP-7).
- **CAP-4**
  - **intent:** The review workflow (SPEC-article-review: lint → structure → prose → cold read, capped severity-tagged findings) runs as a plugin skill against a draft file in the host repo.
  - **success:** Each pass runs once per draft version and emits findings, not rewrites, per that spec's contract.
- **CAP-5**
  - **intent:** The four article frameworks (SPEC-article-frameworks) ship as plugin assets the pipeline fills verbatim.
  - **success:** A produced draft's structure matches the chosen framework template slot-for-slot.
- **CAP-6**
  - **intent:** Owner identity (name, site URL, pointer-block content, canonical/syndication policy, frontmatter schema of the target site) lives in a user config file, not in skill prompts, so the engine is generic.
  - **success:** Deleting the owner's config and supplying a different user's config produces drafts carrying the new identity with zero skill-file edits; grepping the skills for `tim-nish.dev` returns nothing.

## Constraints

- Must run inside any host repository without assuming the website repo's layout; all host-specific knowledge enters via `writing-sources.yaml` (machine-global per-repo config, amended 2026-07-15 per #211 — see the footprint invariant) and the user config.
- Draft output location is declared per host repo in `writing-sources.yaml`; the pipeline never assumes a fixed `drafts/` path. `output.drafts` may name a destination **outside** the host repo (amended 2026-07-15 per #211) — an external private articles repository is the recommended default, so article products (private assets) never enter a maybe-public host repo's history.
- Publication boundary (added 2026-07-15 per #211): repository boundaries follow publication boundaries — the plugin never causes a private pointer (a `policy_source` path, a private article destination, machine-local layout) to be committed into a host repository, because a host repo may be public or become public later and removal costs history scrubbing. Everything carrying such pointers lives in machine-local configuration.
- This repo is self-contained: the three adopted article specs are vendored under `specs/`, the canonical home for implementation specs. Changes to the adopted specs are made in the vendored copies here; the website-repo originals are superseded for this project.
- The three article specs are adopted contracts: this repo ports them to plugin form but does not redesign their internals; behavioral changes go through their own spec updates first.
- No JavaScript/TypeScript anywhere in the plugin: skills, commands, and templates are Markdown; helper scripts are Python (stdlib-only — host repos guarantee no venv or installed dependencies) or POSIX shell. Invoking `npx bmad-method install` as an external tool does not violate this.
- The repository is BMAD-managed from day one (specs + epics + stories), since the owner implements via BMAD.
- BMAD artifacts and hand-written content stay strictly separated: BMAD's footprint is exactly `_bmad/`, `_bmad-output/`, and `.claude/skills/bmad-*`; project specs live only under `specs/`. Release stripping is then a mechanical removal of those three paths with no judgment calls — nothing hand-written may ever land in a BMAD directory, and no BMAD output may land in `specs/`.
- Dogfooding gate: OSS release decisions wait until the plugin has produced real published articles for research-notes/QuantScenarioBench; releasing is a separate later decision, not part of this build. If working-note history must not ship, release happens via a fresh public repo (or squashed export), keeping this BMAD-managed repo private.
- **Target-repo selection** (added 2026-07-17, #309): every entry flow resolves the host repository through one rule — explicit `--root` wins unconditionally, else the git top-level of the current directory, else **fail closed** (exit 2 with a pass-`--root` diagnostic; never a silent cwd fallback) — with both branches realpath-canonicalized, and exactly **one resolved root per run**: no script step (entry gates included) may consult raw process cwd as a side channel. The resolved target path appears in each flow's **first owner-visible output**, before any scope read, workspace mint, or LLM spend; when an explicit `--root` and cwd's git top-level both resolve and differ, a one-line informational notice names both (explicit still wins, fail-open).
- **Host-repo footprint invariant** (engine-wide; added 2026-07-11 from dogfood findings, `docs/storage-architecture.md` D1–D2): the plugin never writes state or intermediate artifacts into the host repository's working tree — the only files it creates there are declared products at `output.drafts`. All storage paths (config lookup, state root, per-run workspaces) resolve through a single stdlib-Python path-resolver helper; no other script, skill, or prompt carries a storage-path literal, so the layout behind the resolver is an implementation detail with one migration point. **The O1 exemption is removed (amended 2026-07-15, #211 — `docs/storage-architecture.md` O1 resolved to machine-global per-repo config):** `writing-sources.yaml` resolves from `~/.config/writing-assistant/repos/<repo-key>/` through the resolver, no in-repo file exists, and the invariant is now total — the plugin writes nothing into a host repository except declared products at `output.drafts`, and `output.drafts` itself may point outside the host. Article plans at `plans/<slug>.md` in the articles repository are a declared product of that destination (added 2026-07-17, #310, SPEC-article-plan) — never a file in the host source repo. During migration, a legacy in-repo `writing-sources.yaml` is still read (machine-global wins when both exist) with a one-line deprecation pointer to the migration path — never silently, never forever. **The destination repo's permitted write surface is named exhaustively (amended 2026-07-22, #550):** at the `output.drafts` **destination** repo — which the lint and footprint check name explicitly, never "host repo" — the plugin writes the **two gated products** (canonical draft, article plan) **plus exactly one regenerated, non-gating view** (`INDEX.md`, SPEC-article-draft-pipeline). Nothing else, and a write outside that set is a footprint violation the check names. **Working-note renderings are ordinary `output.drafts` variant files (corrected 2026-07-24, #658 — the earlier same-day #653 amendment overreached and is withdrawn):** a working-note's email and web-archive renderings are variant files landing at `output.drafts` like every platform variant (already permitted — `SPEC-platform-variants`: "only the variant files land at the `output.drafts` destination"), **not** a new destination product. The plugin does **not** write the published **`newsletter/` section**: that section is an owner publish target — `pipeline-proposes, owner-commits`, lowering is a human act (`consulted: product-lab@90877fa4e77e1353b527a76607ed2ea06daf2b27 topics/articles.md:52,60`) — and a profile's `newsletter/` `layout.dir` is a conformance record of that target (like Zenn's `articles/`), never a plugin write location. The permitted destination write surface is therefore **unchanged** from #550 above. This does not loosen the host-repo invariant above (a host *source* repo still receives nothing), and it does not reach the owner's **site tree**, which the pipeline never writes — `pipeline-proposes, owner-commits` is site policy, and the destination repo is not the site. **A test-declared run never writes to a destination it does not own (amended 2026-07-22, #573; predicate corrected same day at re-triage):** a run that declares itself a test — the `--allow-external-draft` check-harness escape — is refused when the resolved `output.drafts` lies **outside the system temporary directory**, i.e. outside the run's own disposable tree; the run supplies an explicit `--out`, or it aborts naming the destination it refused. A destination **inside** that tree resolves normally. The predicate is the **destination**, not the invocation shape: the first formulation refused config resolution for any test-declared run, which also made the #213 `--create-out` consent path untestable (`scripts/check-stage5-variants.sh:190-211` must resolve a config destination with no `--out` — that *is* what it tests). **Known limit, accepted deliberately:** "under the system temp dir" is an OS path convention, so a harness using a non-temp scratch directory is refused for a reason unrelated to the leak; fixtures live under the temp tree. The `output.drafts` destination is a **machine-wide** resource that reads as a per-run one, so it is isolated rather than shared silently (`policy: a shared-scope resource that presents as private must be isolated OR warn loudly on collision — silence is not among the options`, product-lab `topics/claude-code-ops.md:9`). The observed leak path is not a temp host — every temp host resolves its own tree — but the **plugin's own development repo**, which is a configured host whose destination is the owner's real articles repo, so an unrooted test invocation from it writes a real canonical.
- **Owner-facing proposal contract** (engine-wide; added 2026-07-10 from dogfood findings): every prompt that asks the owner to approve, modify, or decline something — gap-interview questions, owner-verification items, review-arbitration findings, visual proposals, and any future proposal surface — must show (a) **where** the item lands in the artifact (outline/section context, with a short preview of the current content when one exists), (b) **why** it is being asked, and (c) choices whose labels state their **concrete effect on the artifact** — never shorthand labels that require inferring the generation logic. A first-time user must be able to answer from repository knowledge alone, without already understanding the generated draft. Additionally (added 2026-07-11, `docs/interview-architecture.md` D4): (d) **selective presentation is the primary interaction model** — every proposal surface presents choice-based selective prompts, with free-form text entry appearing only as the input mode for owner-only knowledge (open interview questions, modify/replace paths); collecting answers as free-form text where choices are mandated is a contract violation, not a presentation preference; and (e) **payload integrity is validated before presentation** — every item's where/why/effect fields present, non-empty, and untruncated, with content fitting its display budget by authorship (re-written shorter), never by clipping; a validation failure blocks presentation the way `verify-markers` blocks stage progression. Stage specs reference this contract; they do not restate it.
- **Enforced-mechanism invariant** (engine-wide; added 2026-07-22 from triage #534/#533): any **budget cap, cache, or checkpoint** the engine *promises* is enforced by a **mechanical assertion**, never by per-prompt compliance alone — caps and caches are architecture, not prompt hygiene. A run that reaches its completion summary while a promised mechanism did **not** execute (e.g. a completed harvest whose blob-keyed cache was never populated, #534) **fails loudly and names the omission** — a silently-skipped mechanism that leaves a valid-looking product is a defect, not a warning. Correspondingly, **each interruptible stage declares its interruption/resume contract**: the state it checkpoints, where a resume re-enters, and that the budget-triage **orderly stop** (SPEC-article-draft-pipeline durability constraint) binds **identically on resumed runs** and **discloses re-done work before spending** it — an interruptible stage without a stated resume contract is the same class of gap as an uncapped fan-out. Harvest states its contract first (SPEC-article-draft-pipeline CAP-10 + durability constraint); other stages state theirs as they are hardened. The mechanical assertion enforcing a promise is authored **in lockstep** with the prompt/skill step that describes it — a change to one without the other is a defect (the same lockstep discipline the fact-sheet KIND set carries).
- **Gate-item content grounding** (engine-wide; added 2026-07-22 from triage #567, ratifying #526's fix as a general property): the owner-facing proposal contract above binds gate-item **content**, not only its form. **A machine-authored gate item asserts no factual premise without a resolvable pointer, or an inline `unverified —` marker at the point of use.** A well-formed option carrying an invented premise passes every structural check and still corrupts the ratification, because the owner ratifies the *content* — the failure mode is an owner skimming a plausible "described internally as…" clause and ratifying an invention, so the judgment the gate captures is about a world that does not exist. Three properties are load-bearing: (a) the **marker spelling is `unverified —`, inline, at the point of use** — never a footnote the reader may skip, and deliberately the same spelling adopted for unverified operational references, decided once across both surfaces rather than twice; (b) the rule attaches to **premise clauses specifically, not to an item's overall honesty** — #526's item disclosed correctly at the top level (`not in declared sources`) and still smuggled a fabrication into a subordinate clause, so top-level candour is **not a defence**, and an honest top-level disclosure can actively make the invention read as licensed; (c) a **pointer-carrying** premise validates under the same resolvable `file:line@commit` grammar as an evidence entry. This binds **every** machine-authored gate item — NEEDS-OWNER items, interview items, fork-gate candidates and FYIs, proposal payloads, review-arbitration findings, visual proposals, and any future proposal surface — and is the third artifact class of one rule (spec bindings → reported references → gate premises). Per-surface specs state their **instance** of this constraint and its validator; they do not restate the rule, and they may not narrow it.

## Non-goals

- No publishing automation (dev.to/Zenn APIs) — the owner publishes, per SPEC-article-draft-pipeline.
- No topic selection or idea backlog.
- No web UI or service component — Claude Code plugin only.
- No multi-user/team features in v1; single-owner config.
- No BMAD workflow meta-commands in the plugin surface: `/bmad-setup` and `/bmad-epics` are owner-workflow tooling and live in the owner's global `~/.claude/commands`, not in this repo's capabilities.

## Success signal

- After research-notes exists, the owner installs the plugin there, runs the pipeline on real material, and hands a schema-valid draft to review in one session — the first dogfooded article that never touched the website repo's tooling.

## Assumptions

- Claude Code's plugin mechanism (`.claude-plugin` layout, repo-as-marketplace) is the distribution vehicle; during development the same content runs as plain local skills/commands (`claude --plugin-dir`, or symlinked into a host repo's `.claude/`), so packaging is additive and no submodule/`npx` bootstrap fallback is anticipated.
- Cross-repo harvest reads sibling checkouts on local disk (paths in `writing-sources.yaml`); the only sanctioned GitHub API read is a declared `github-issues` source (amended 2026-07-17, #301) — read-only, label-filterable, host-repo issues only; no other API scraping exists.
